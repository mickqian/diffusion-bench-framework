Serving-first design

Diffusion performance is not a single kernel story. SGLang-Diffusion exposes the production controls that decide whether image and video generation can be served predictably at scale.

01

Native pipeline ownership

Stages, tensors, schedulers, encoders, denoisers, and decoders are represented directly instead of wrapped as opaque scripts.

02

Batching and placement

Dynamic batching, performance mode, and layerwise offload keep GPU residency explicit under real traffic.

03

Parallel execution

TP, SP, Ulysses/Ring, and CFG parallel paths are tracked per model and hardware profile instead of guessed at runtime.

04

Measured tradeoffs

Approximate acceleration is opt-in and paired with quality drift, artifact checks, and customer-facing acceptance thresholds.

Model surface

One serving layer for image and video.

The benchmark framework tracks command profiles, versions, and failure reasons so coverage expands without losing reproducibility.

FLUX.1 / FLUX.2 Qwen-Image Z-Image Cosmos3 Wan 2.1 / 2.2 LTX 2 / 2.3

Text-to-image

FLUX, Qwen-Image, Z-Image, Cosmos3, and other image families with tracked single-GPU and TP profiles.

Text-to-video

Wan, Cosmos3, and LTX style workloads where sequence length, decoder cost, and CFG parallelism dominate.

Image editing

Editing pipelines reuse the same serving controls while preserving model-specific prompts, image inputs, and recipes.

Framework tracking

Best commands are versioned per framework and hardware, including SGLang-Diffusion, vLLM-Omni, LightX2V, and diffusers baselines.

What users get

The goal is not only faster demos. The goal is a repeatable serving surface where model teams can ship, compare, and tune diffusion workloads without rebuilding the runtime per model.

Cross-framework truth

Formal reports keep raw data, exact commands, framework versions, hardware, and benchmark policy together.

Production controls

Serving knobs are first-class: dynamic batching, offload, parallelism, attention backend, compile, and postprocess placement.

Quality-aware speed

Lossy paths such as cache, quantization, or progressive resolution are evaluated as speed-quality frontiers.
